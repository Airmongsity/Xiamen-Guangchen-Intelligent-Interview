# AutoMemory

A **human-like long-term memory module** for AI agents. Whether a memory survives and how highly it ranks at recall time is determined by four criteria: **time decay, prior importance, retrieval frequency, and outcome feedback**. Usable as a Python library or as an MCP server for autonomous tool use by an AI agent.

> **Note on authorship**: this is a *vibe coding* project — all code was generated by Claude (Fable 5 & Opus 4.8). My own contributions are limited to `plan.md` and this `README.md`.

## Why another memory system?

Among the three popular engineered open-source memory projects — [mem0](https://github.com/mem0ai/mem0), [MemGPT](https://github.com/letta-ai/letta), and [PowerMem](https://github.com/oceanbase/powermem) — an **explicit outcome-feedback loop is missing**: once a memory is stored, its "importance" rests forever on a prior guess. AutoMemory exposes this loop as a clean library-level API: `recall()` returns a `retrieval_id`, and the caller reports back with `report_outcome(retrieval_id, quality)`. Memories that empirically helped rank higher and decay more slowly; memories that misled are forgotten faster (idempotent, rank-discounted updates).

This is a **differentiated engineering design relative to mainstream open-source implementations, not a novel mechanism**. The idea has academic precedent: RMM ([Tan et al., ACL 2025, arXiv:2503.08026](https://arxiv.org/abs/2503.08026)) has the LLM assign ±1 citation scores to retrieved memories based on their utility in the response and uses them as RL rewards to update a reranker, reporting >10% accuracy improvement on LongMemEval over a no-memory-management baseline. AutoMemory implements the same intuition as per-memory scalar utilities updated by caller feedback, rather than learned reranker weights.

## How it works

Human memory is not a key-value database — it is a system that *forgets, is reinforced by use, and recalls by cued association*. AutoMemory models it accordingly:

1. **Short-term memory (STM)**: the most recent raw messages are provided in full (last 20 messages ∩ 6-hour TTL). Messages that overflow the window are consolidated into long-term memories via `consolidate()`.
2. **Long-term memory (LTM)**: self-contained facts / experiences / summaries extracted by an LLM. Each memory carries `importance` (prior), `utility` (empirical), `access_count` (review strength), and timestamps.
3. **Forgetting curve**: `retention = exp(-Δt / S_eff)`. Being retrieved counts as a "review" (refreshes the decay anchor). Important, useful, and frequently recalled memories decay more slowly.
4. **Hybrid retrieval**: vector semantic recall ∪ BM25 lexical recall → combined scoring → single-hop activation spreading along logical links (`related` / `derived_from` / `contradicts`) → neural reranking → top-k.
5. **Outcome feedback**: after the retrieved results are used, the caller reports quality; each memory's `utility` is updated with a DCG-style rank discount, closing the loop on both ranking and decay.

Writes are not blind appends: a new memory is first compared against its semantic neighbors (semantic-KNN ∪ same-attribute `slot` lookup, so a value change like "works at Acme" → "works at Globex" is caught even when the wording barely overlaps), and an LLM decides **ADD / UPDATE (merge) / DELETE (contradiction) / NONE (no new information: discard)**. Crucially, *nothing is hard-deleted*: UPDATE and DELETE perform **bi-temporal invalidation** — the old value is stamped with a `valid_to` date and kept as retrievable history (`superseded_by` points to its successor, `superseded_penalty` down-weights it), while the new value becomes current. "User now uses Rust" supersedes "user uses Python" for present-tense questions, yet "what did the user use *before*?" is still answerable. A single-valued `slot` is enforced to have exactly one current value.

Each extracted memory also keeps a **pointer to the verbatim passage** it came from (hybrid granularity: compressed fact as the retrieval key, raw text as the value), so recall can attach exact source wording when a question needs names/dates/quotes the extraction smoothed over.

## Scoring mechanics

All parameters live in `ScoringParams` for easy ablation:

```
S_eff     = S_base(kind) × (1 + 0.3·ln(1+access_count)) × (1 + importance + max(utility,0))
            # S_base: fact=14d, experience/summary=30d; source='self' ×1.5
retention = exp(-Δt_days / S_eff)              # Δt anchor = max(created_at, last_accessed)
            # Δt measured from `now`, overridable via recall(as_of=...) to score at query time
relevance = 0.7·cos_norm + 0.3·bm25_sigmoid
priority  = (0.2 + 0.8·retention) × (1 + 0.6·importance + 0.8·utility) × superseded_penalty?
            # 0.2 = decay floor; utility weighted above importance: evidence > prior
            # superseded_penalty=0.4 applied iff the fact has been invalidated (valid_to set)
score_pre = relevance × priority
            # + single-hop spreading: neighbors gain 0.35·w(m,n)·s_m (≤2 contributors/node, capped at 0.5·s_m)
final     = 0.7·rerank_score + 0.3·minmax(score_pre)
feedback  : utility ← utility + 0.3·(1/log2(rank+2))·(2·quality−1 − utility)   # DCG discount + EMA, idempotent
```

## Get started

```python
from automem import AutoMemory, AutoMemConfig

am = AutoMemory(AutoMemConfig.from_env())

# Write: conversation extraction / AI self-notes / batch import / short-term buffer
am.add([{"role": "user", "content": "My name is Ming and Python is my favorite language"}])
am.remember("Lesson: always run tests before committing", kind="experience")  # source='self'
am.add_batch([{"content": "A historical fact", "created_at": "2025-01-01T00:00:00+00:00"}])
am.observe("user", "the message just now")  # STM buffer only, zero latency
am.consolidate()                            # consolidate overflowed STM into LTM

# Recall → inject into context → report outcome
result = am.recall("What language does the user prefer?")
print(result.to_prompt())                   # injectable text, carries retrieval_id
am.report_outcome(retrieval_id=result.retrieval_id, quality=1.0)

am.maintain()                               # soft forgetting: retention < 0.05 and no positive utility
```

## Evaluation results

> **Setup & comparability disclaimer**: all numbers were produced with real API calls, using **DeepSeek as both answerer and LLM-judge**. They are therefore **not directly comparable to public leaderboard numbers**, which typically use GPT-4o as judge. They are internal controlled comparisons against two baselines run under *identical* conditions — same questions, same reader prompt, same judge: *No Memory* (answer blind) and *Naive RAG* (raw conversation chunks, same embedding model, same top_k=30).

### LongMemEval-S — 150-question held-out (the headline)

This is the **only headline table**: a 150-question stratified held-out set, evaluated **once** with the shipped config, never used for tuning. (A separate 60-question dev set was used during development — see [the dev-vs-held-out gap](#the-dev-set-was-optimistic--held-out-caught-it), which is itself the point of holding a set out.) Each question hides its evidence inside ~50 sessions / ~122k tokens of distractor history; retrieval is scored **read-only** and **as-of the question date** (see [Evaluation integrity](#evaluation-integrity)).

> **⚠️ Pre-prompt-rewrite numbers.** This table (AutoMemory **67.3%**) was produced with the original terse extraction prompt. The extractor has since been rewritten to preserve verbatim specifics — proper nouns, exact numbers, qualifiers — directly targeting the *compression* gap the Recall@k panel localizes below. On a controlled 60-q dev A/B (same questions, same judge) that rewrite scored **+3.3pp overall (66.7% → 70.0%)**, with the gains where compression hurt most (**multi-session +23pp**, single-session-user +14pp). The 150-q held-out has **not** yet been re-run, so the numbers in this section still reflect the old prompt; the new headline is **pending a held-out re-run**.

| multi-session | 37 | 0% | 56.8% | 56.8% |

**Retrieval quality (session-level Recall@k vs the evidence labels) — the key diagnostic:**

| | AutoMemory | Naive RAG |
|---|---:|---:|
| gold session in top-k (hit rate) | **0.993** | **0.993** |
| mean fraction of gold sessions covered | 0.951 | 0.984 |

Honest takeaways:

- **Roughly at parity, slightly behind raw RAG: 67.3% vs 70.7%** (~5 questions on 150). Big lift over no-memory (7.3% → 67.3%) confirms the benchmark genuinely tests evidence-finding.
- **The gap is compression, not retrieval — and Recall@k proves it.** Both systems put the gold session in the top-k **99.3%** of the time, so AutoMemory almost never *fails to retrieve*; it loses where, having retrieved the right session, its *compressed* memories drop the verbatim detail that RAG's *raw turns* keep. The damage is exactly in the wording-sensitive categories — single-session-assistant (76.5% vs 94.1%, retrieved-but-answered-wrong), preference, user.
- **AutoMemory's one genuine advantage is temporal-reasoning** (55.3% vs 47.4%, n=38 — the only category where it leads). Modest in absolute terms (~3 questions) but directionally consistent with the bi-temporal `valid_from`/`valid_to` + as-of machinery it is built around.
- **Multi-session is a tie at this scale** (56.8% each, n=37). An earlier dev-set run had suggested a large AutoMemory lead here; it did not survive (see below). The hypothesis that extraction's compression *should* win as histories outgrow what RAG can index wholesale is **unverified at LongMemEval-S scale** and is left as a longer-horizon (BEAM-scale) question, not a current claim.

### Ablation: which mechanisms earn their place (60-q dev set)

Measured on the 60-question dev set (so these are mechanism contributions, not held-out scores). Each row removes/changes **one** thing, holding the rest fixed under the clean harness:

| Configuration | Overall (dev) | Δ |
|---|---:|---:|
| **Full** (bi-temporal history competing in-pool + as-of retention) | **80.0%** | — |
| − bi-temporal history (retrieve current values only) | 70.0% | **−10.0** |
| − as-of retention (score decay at wall-clock now, not question date) | 70.0% | **−10.0** |
| history segregated into a separate quota instead of competing | 73.3% | **−6.7** |

- **Bi-temporal history pays for itself (+10 pts on dev).** Not deleting superseded facts — keeping them retrievable, down-weighted (`superseded_penalty=0.4`), *mingled into the ranked evidence* — feeds the timeline reasoning that is AutoMemory's one held-out advantage (temporal-reasoning); the old values are real evidence for "before"/"how long between" questions.
- **As-of retention matters (+10 pts).** Scoring the forgetting curve at the moment the question was asked (rather than wall-clock) is what lets recency discriminate at all on archived conversations; without it retention collapses to a constant and the priority machinery goes inert.
- **Segregating history into its own quota was a net loss (−6.7).** An earlier refactor moved superseded facts out of the top-k pool into a separate "earlier values" appendix to stop them diluting single-turn recall. Measured cleanly it *hurt*: the reader under-uses an appendix, and for multi-session synthesis historical facts deserve to be first-class (just down-weighted) evidence. It is kept as a config toggle (`history_separate_quota`, with a query-aware gate) but is **off by default**.

The runner is now fixed in code, but these figures were **not** re-run — treat them as **qualitative only** (direction, not magnitude), not as a calibrated comparison.

### Evaluation integrity

The numbers above are the *corrected* ones. Reaching them surfaced two measurement bugs that had silently **inflated and then scrambled** earlier results — worth documenting, because both are easy traps for any memory-system benchmark.

1. **The evaluation was mutating the thing it measured.** `recall()` defaults to `touch=True` (a retrieval reinforces a memory — refreshes its decay anchor and bumps `access_count`) and `log=True`. Run live, that is the point of the system. Run as a benchmark — replaying the *same* questions over the *same* store across many configurations — it means each pass leaves the store stronger on exactly the memories it just surfaced. Because the eval data is backfilled to 2023 while wall-clock was 2026, an untouched memory had retention ≈ 0 (priority floored) but a *touched* one jumped to retention ≈ 1 — a 5× priority swing. So evidence retrieved in run *N* was boosted into run *N+1*: **relevance leaking across runs**, monotonically inflating scores (we watched a config climb from .717 to .783 purely by re-running it). The fix: `recall(touch=False, log=False)` for evaluation, a one-line read-only path; reset the polluted columns; and a regression test (`test_recall_readonly_does_not_mutate_state`) that fails if a read-only recall ever writes state again.

2. **The forgetting curve was anchored to wall-clock, not to the question.** With `now` = 2026 and memories from 2023, every retention collapsed to ≈ 0 *uniformly*, so the entire decay/priority mechanism contributed nothing to ranking — it was dead weight, and (perversely) the touch-pollution above was the only thing reviving it. The fix: `recall(as_of=<question_date>)` scores the curve at the moment the question was actually asked. The ablation table credits this with +10 points — i.e. the priority machinery only does measurable work once it is timed correctly.

How the corrected harness is now protected:

- **Read-only by assertion**: evaluation calls `recall(touch=False, log=False)`; a canary test locks this in.
- **As-of enforced**: the search runner passes each question's date as `as_of`; the decay anchor is the ask-time, not the run-time.
- **Deterministic & comparable**: with no state mutation and a fixed `as_of`, retrieval is identical across runs, so a re-judge reproduces a config's score exactly (verified) — config deltas are signal, not drift.

The broader lesson: a memory system's "recall reinforces memory" feature and a benchmark's "measure the same store repeatedly" assumption are in direct conflict. Any A/B over a shared store must be read-only, and any replay of historical conversations must score time as-of the conversation.

#### The dev set was optimistic — held-out caught it

The 60-question dev set was used throughout development: configs were chosen on it, and the shipped default is the one that scored best on it. That makes its number an **optimistic in-sample estimate** by construction. The 150-question held-out run — evaluated once, never tuned against — is what the dev number should be checked against, and the gap between them is precisely *why* a set is held out:

| | dev (60-q, in-sample) | held-out (150-q) |
|---|---:|---:|
| AutoMemory overall | 80.0% | **67.3%** |
| Naive RAG overall | 78.3% | **70.7%** |
| AutoMemory − RAG | +1.7 | **−3.4** |
| multi-session (AutoMem vs RAG) | 92.3% vs 53.8% (+38.5) | **56.8% vs 56.8% (tie)** |

Two distinct effects inflated the dev reading, and the held-out set exposed both:

1. **Selection on the dev set.** Picking the best-scoring config on the same 60 questions you report bends the number upward. The held-out run removes that coupling.
2. **Small per-category n.** The flagship "multi-session +38.5" claim rested on **13** questions; at **37** held-out questions it is a flat tie. Category winners on n≈4–16 are mostly sampling noise — which is why the held-out table leads with n and why no single-category claim should be read as decisive below ~n=30.

The honest correction is symmetric — guarding against false *pessimism* as much as false optimism: the held-out result is not a failure but a **parity-minus result with equal retrieval and one real (temporal) edge**, and the larger sample makes it more trustworthy than the dev number in *either* direction. The Recall@k panel is what keeps the diagnosis grounded: it localizes the remaining gap to compression, not retrieval, rather than leaving us to speculate.

## Strengths and weaknesses

### Strengths

- **Retrieval on par with raw RAG at a fraction of the footprint**: held-out session-level Recall@k is **0.993, identical to RAG's**, while storing ~320 compressed memories/question vs RAG's ~494 raw chunks — it finds the right evidence in massive distractor history just as reliably, with less to index.
- **A real edge in temporal reasoning**: the one held-out category where AutoMemory leads RAG (55.3% vs 47.4%), driven by the bi-temporal timeline machinery below.
- **Bi-temporal knowledge updates**: superseded facts are invalidated (`valid_to`/`superseded_by`), not deleted — present-tense questions see the current value, "what was it before" stays answerable, and the retained history is +10 points of measured benefit on dev (see ablation). Single-valued attributes carry a `slot` so value changes are caught and enforced to one current value.
- **Hybrid-granularity recall**: each fact points back to its verbatim source passage; recall attaches the raw text so exact names/dates/quotes partly survive the compression — the lever for the verbatim-recall gap.
- **Explicit outcome-feedback loop**: `report_outcome` lets a memory's importance be falsified/confirmed by downstream effects rather than resting on priors alone.
- **Human-like forgetting + review reinforcement**: Ebbinghaus decay + retrieval-as-review keeps stale memories sinking and frequently used ones evergreen; `maintain()` soft-forgetting bounds store growth. Retention is scoreable as-of any moment (`recall(as_of=...)`).
- **Storage compression**: ~half the volume of raw conversation after extraction.
- **Engineering robustness**: single-file SQLite, zero deployment; sqlite-vec auto-fallback to numpy; thread-safe (thread-local connections + WAL); provider clients retry 6× against 429; truncated-JSON extraction is salvaged and per-session ingest failures are isolated; evaluation runners resume from checkpoints and run read-only. 88 tests (85 offline + 3 online smoke).
- **Ablation-ready**: every scoring parameter lives in `ScoringParams`; feedback / spreading / decay / bi-temporal history / as-of timing can each be toggled independently.

### Weaknesses and known boundaries

- **Behind naive RAG overall on held-out: 67.3% vs 70.7%.** Not ahead. Retrieval is equal (Recall@k 0.993 each), so the deficit is the *compression tax* — extracted memories drop verbatim detail that raw turns keep. The hypothesis that compression wins as histories outgrow what RAG can index wholesale is unverified at this scale; on LongMemEval-S, raw RAG is the stronger baseline.
- **Extraction loses verbatim detail**: the losses concentrate in wording-sensitive categories — single-session-assistant (76.5% vs RAG 94.1%, *retrieved but answered wrong*), preference, user — where the answer is one exact line that raw-chunk RAG reproduces and extraction smooths over. Hybrid-granularity source pointers close part of this gap but not all of it.
- **Recall needs retuning as the store grows**: a fixed top_k dilutes recall as extraction gets richer. The separate-history-quota refactor that tried to address this was measured to *hurt* and is off by default; adaptive / token-budget recall is the open direction.
- **The feedback loop's benefit is not yet proven by benchmarks**: in ablation its gain falls within noise — LongMemEval/LoCoMo are one-shot QA with no "same memory repeatedly retrieved and reported" pattern, so the loop never spins up. Validating it requires long-session / repeated-query workloads or online A/B.
- **Extraction/rerank call cost**: one LLM extraction per session makes ingestion slow (~80 min to ingest the 60-question haystacks). Suits write-light/read-heavy long-term assistants, not high-throughput ingestion.
- **Provider lock-in**: currently hard-wired to DeepSeek + SiliconFlow (both OpenAI-compatible); switching requires edits in `providers/`.
- **Evaluation scale**: a 150-question held-out set (+ 60-q dev), DeepSeek-judged, not the full 500 with a GPT-4o judge. Per-category n is 7–38, so single-category claims below ~n=30 are noisy; the overall and the larger categories (temporal n=38, multi-session n=37) are the reliable signal.

## Roadmap

*Done since the first milestone* (were roadmap items, now shipped and ablated above): **hybrid-granularity storage** (fact → verbatim source pointer) and **bi-temporal invalidation instead of DELETE** (`valid_to`/`superseded_by`, history retained and competing).

- **Time-constrained retrieval** for temporal-reasoning: extract time windows from the query and filter before semantic recall — the largest remaining verbatim deficit.
- **Token-budget / adaptive recall** instead of a fixed top_k, so recall does not dilute as the store grows.
- **Full-scale comparable evaluation**: all 500 LongMemEval questions with a GPT-4o judge, plus integration into [mem0's open memory-benchmarks suite](https://github.com/mem0ai/memory-benchmarks) for a true same-condition comparison. (Session-level Recall@k against the evidence labels is **now reported** on the 150-q held-out — it already separates retrieval misses from reasoning errors, and pinned the current gap on compression, not retrieval.)
- **A feedback-exercising benchmark**: repeated-query workloads, or RMM-style automatic citation scoring to replace reliance on manual `report_outcome` calls.

## Reproducing the evaluation

```bash
# ---- LongMemEval-S (primary): the full held-out run, one command ----
python -c "from huggingface_hub import hf_hub_download as h; h('xiaowu0162/longmemeval','longmemeval_s',repo_type='dataset',local_dir='evaluation/longmemeval/data')"
cd evaluation/longmemeval
python run_full.py --tag lme_full --limit 150 --workers 8 --rag-workers 2   # ingest -> automem -> rag -> none -> judge x3
python summarize.py --tag lme_full                                          # 3-system accuracy + session-level Recall@k
        # run_full ingests once (session-tagged, --resume-able), then evaluates all three systems on the SAME
        # manifest with the SAME reader prompt + judge. search is read-only and scores retention as-of the
        # question date; history competes in-pool (shipped config). search_runner ablation flags:
        #   --no-asof  (wall-clock retention) | --separate-quota (segregated history + --history-mode auto|on|off)
        # each run drops a {tag}_{system}_config.json recording reconcile_floor / as_of / quota for provenance.

# ---- LoCoMo (counter-reference) ----
cd ../locomo
python add_runner.py    --db ../results/locomo.db --conv-ids 0
python search_runner.py --db ../results/locomo.db --conv-ids 0 --out ../results/automem_results.json
python judge.py --input ../results/automem_results.json
python ../ablation_feedback.py --db ../results/locomo.db --conv-ids 0 --variants full,no_feedback,no_spread,no_decay
```

> Note: the evaluation runners give each question its own `user_id` namespace (per-question haystacks stay isolated). `stats()` filters by `user_id='default'` and will therefore show `memories: 0` — this is expected; the data lives under per-question namespaces.
