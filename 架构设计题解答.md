# 架构设计题 · 解答（Joey Chan）

> 这是 2026 Agent 技术笔试 **第二部分：架构设计题** 的解答（第一部分 Vibe Coding 的代码/README/录屏见仓库其余内容）。
> 共 5 个模块，**每模块选 1 题作答**。每题结构：**方案 → 关键取舍 → 优缺 → 我的选择**，并尽量挂钩我的真实产出（自研记忆系统 [AutoMemory](https://github.com/Airmongsity/AutoMemory) 的实测数据 / 本仓库 `mini_agent` 代码）。

## 模块一 · Context/Performance —— Q2：一个 session 连续聊了 200 轮，context 快爆了，怎么压缩？如何确保压缩后仍流畅？

**方案（分层，不是一刀切摘要）**：近 N 轮保留**原文**（模型对最近上下文最敏感）；溢出的旧轮由 LLM **递归摘要**成滚动 summary；稳定事实（偏好/决定/未决项）抽进**长期记忆**，需要时再召回，而不是永远堆在 context。

**保证“流畅”的工程细节**：摘要**强制保留 实体 / 数字 / 日期 / 未决请求（open threads）**——流畅性主要死在“忘了用户上一件没做完的事”；**只在 user 轮边界切**，绝不切断 assistant 的 `tool_calls` 与其 `tool` 结果（否则 OpenAI 兼容 API 直接报错），这正是我在 `mini_agent/context_manager.py::_safe_split` 的实现；对 names/quotes 等**逐字敏感**内容保留**原文指针**，召回时贴回。

**取舍与实测**：纯摘要省 token 但丢细节；纯 RAG 保原文但挤占 context。我在 AutoMemory 上做过对照——LongMemEval-S 上压缩版 **67.3%** vs raw RAG **70.7%**，而两者 session Recall@k 都是 **0.993**。**差距不是“检索不到”，而是“压缩丢了逐字细节”。**

**我的选择**：滑窗 + 递归摘要（保 open threads/实体）为主，逐字敏感项挂原文指针，长期事实外置到 memory；复杂分层压缩非必要不做。

## 模块二 · Memory —— Q1：和聊天 Agent 熟悉半个月后，用户问了一个以前问过的问题，Agent 如何做 memory 召回更合理？

**方案（即我的 AutoMemory 召回链路）**：混合检索（向量 ∪ BM25）→ 打分（`priority = (0.2 + 0.8·retention)·(1+…)`，`retention = exp(-Δt/S_eff)`）→ 单跳联想扩散 → rerank → top-k。

**“重复问”的关键闭环**：第一次问过 = 该记忆被**检索即复习**（刷新衰减锚点、`access_count+1`），且被 `report_outcome` 正反馈过 utility → **第二次召回排更高、命中更快更稳**。这就是“越用越熟”的机制。此外按“提问时刻”算衰减（as_of），而非墙钟，否则旧对话 retention 会塌成常数。

**landscape 与差异化（顺带答 Q2）**：经典框架 mem0 / MemGPT(Letta) / PowerMem；趋势是**加显式 outcome-feedback / RL reranker**（RMM, ACL 2025，>10% 提升）。**主流开源记忆框架普遍缺显式效用反馈闭环——我在 AutoMemory 补了这一环**（recall 返回 retrieval_id，用完 `report_outcome` 更新 utility 与衰减）。

**诚实边界（自我评估）**：我在 held-out 上总体仍略低于 raw RAG，瓶颈是压缩而非检索（Recall@k 持平）——我更愿意讲清边界而不是夸大。

**我的选择**：混合检索 + 遗忘曲线 + 检索即复习 + 效用反馈；重复问场景靠“复习 + 正反馈”自然收敛。

## 模块三 · Task —— Q2：用户给 Agent 下达任务“每天早上 9 点根据昨天聊天情况做复盘总结”，你会怎么设计？

**方案（调度层 + 复用记忆构件）**：① cron/scheduler 按用户时区每天 09:00 触发；② 拉取该用户昨日会话；③ 走 AutoMemory `consolidate()` 把昨日内容抽成 **summary 记忆**，再 LLM 生成结构化复盘（做了什么 / 进展 / 未决 / 今日建议）；④ 复盘**写回长期记忆**（供后续召回）并推送给用户。

**关键工程点（易被追问）**：**幂等**（同一天只跑一次，dedup key = user + date）；**失败重试 / 补偿**（09:00 挂了要能补跑）；**时区**要对；**昨日无对话**要优雅退化。**边界意识**：AutoMemory 已有 `consolidate()`/summary 构件，但“定时调度”属 runtime/任务层、需新加。

**取舍**：纯定时脚本（简单、独立）vs 把“复盘”做成 agent 的异步任务（可与模块四的异步工具/通知统一）。

**我的选择**：独立 scheduler 触发，归纳复用 AutoMemory，结果回灌记忆闭环；先把幂等 / 时区 / 空数据这几个必翻车点兜住。

## 模块四 · Tool / Session Runtime —— Q2：session state 为 busy 时，用户又发来新消息，或异步工具完成事件也到达，runtime 应如何处理？

**方案（单会话串行化 + 事件队列 + 状态机）**：每个 session 一台状态机 `idle → busy → waiting_async`，**同 session 内串行**以避免历史竞态；一个 per-session 事件队列消费 `user_msg` / `async_tool_done` / `cancel`；busy 时新事件**入队**按序处理；异步完成事件到达时把结果**回灌 context** 唤醒对应轮次继续 loop（对应异步工具的“提交即返 task_id + 完成通知”）。

**busy 时来新用户消息的策略（明确给出并说理由）**：**默认排队**（保证一致性，不打断进行中的工具链）；**显式打断才抢占**（用户说“停/改”→ cancel 当前轮、清理半成品、消费新消息）；也可**合并**（把补充信息并入下一次 LLM 输入）。

**取舍**：抢占（响应快、易半成品/竞态）vs 排队（一致、可能让用户等）vs 合并（省一轮、语义可能串）。

**亲身挂钩**：`mini_agent` 现为**单线程同步**，session 已按 id 隔离（`SessionManager` 线程安全 + 每 session 独立历史）；要支持异步，我会在此之上加“每 session 事件队列 + 状态机”，而不引入跨 session 全局锁——这是从 sync 到 async 的最小增量。

**我的选择**：per-session 串行 + 队列 + 状态机；默认排队、显式抢占、可选合并；异步结果经 task_id 回灌唤醒。

## 模块五 · Agent Runtime 架构对比 —— Q1：Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？各自优缺？

> 证据取自两份源码：`claude-code-main/`（Anthropic 侧，通过渠道获取的 Claude Code 源码）与本仓库 `mini_agent/`（我实现的 OpenAI-compatible 侧），行号可坐实。

**机制本质区别**
- **OpenAI-compatible（GLM/豆包/DeepSeek 等）**：工具结果是一条 `tool` 角色消息、`content` 是**纯字符串**；`tool_calls.function.arguments` 是**字符串化 JSON**。见 `mini_agent/runtime.py`（回灌 `{"role":"tool","tool_call_id":…,"content":json.dumps(payload)}`）与 `parser.py::salvage_json`（专门兜底坏/截断 JSON）。
- **Claude Code（Anthropic Messages）**：工具结果是一个 **`tool_result` content block**，放进一条 `user` 消息的 content **数组**里，可与图片等 block 并列；每个工具自实现 `mapToolResultToToolResultBlockParam(content, toolUseID)` 把**强类型 `Output`** 序列化成模型可见 block（`claude-code-main/src/Tool.ts:557`）；带 **`is_error`** 标志把错误**带内**回传（`src/services/tools/toolExecution.ts:1032`）；图片作为顶层 block 与 tool_result 并列实现**同轮多模态**（`toolExecution.ts:1029`）；**并行工具**结果多 block 批在同一条 user 消息（`src/services/compact/sessionMemoryCompact.ts:211`）。

**Claude Code 的关键设计：模型视图 / UI 视图 / 搜索视图三分离**——同一工具输出维护 `mapToolResultToToolResultBlockParam`（喂模型）、`renderToolResultMessage`（终端 UI，`Tool.ts:566`）、`extractSearchText`（搜索索引，`Tool.ts:599`）三种表征；OpenAI-compatible 侧只有一份 `content` 字符串，模型视图 = 日志视图。

**优缺**
- *Anthropic 风格*：结构化 + 多模态 + `is_error` 语义化 + 视图分离 + 天然并行，表达力强、适合复杂/编码 agent；但协议复杂、实现成本高、生态绑定、token 更重。
- *OpenAI-compatible*：通用（国产模型即插即用，我 `mini_agent` 靠这点在 GLM/DeepSeek/Qwen 间随意切）、协议极简、易调试；但结果被压成字符串→多模态/错误语义受限、易出坏 JSON 需自兜底、模型/日志视图混一。

**结论（取舍非站队）**：**Claude Code 把“工具结果”当成一等的结构化多模态对象并区分受众；OpenAI-compatible 把它当成一条字符串换取最大兼容性。** 做深度/多模态/长程 agent 选前者值那份复杂度；要跨多家国产模型快速落地选后者，代价是自己补齐多模态 / 错误语义 / 坏 JSON 兜底这几块。
